# 链路说明

系统主要包含三条业务链路：

1. 用户发起 AI 任务和轮询查询任务状态；
2. 用户注册；
3. 用户登录。

其中，用户发起 AI 任务和轮询查询任务状态是系统的主要业务链路。

## 用户发起 AI 任务和轮询查询任务状态

流程如下：

1. 用户携带 JWT 提交 AI 任务；
2. 后端校验 JWT；
3. 后端进行 IP 限流和用户限流；
4. 后端创建全局唯一的 `task_id`；
5. 后端将任务写入 MySQL，初始状态为 `pending`；
6. 后端将任务消息投递到 MQ；
7. 后端返回 `task_id` 给前端；
(消费端不做具体实现，写出来只是为了连贯性)
8. 消费端消费 MQ 中的任务；
9. 消费端调用 AI 服务；
10. 消费成功后更新 MySQL 中的任务状态和结果；
11. 消费失败且达到最大重试次数后，更新 MySQL 中的任务状态为 `failed`；
12. 消费端将 `task_id -> success/failed` 写入 Redis，并设置过期时间；
(前端定时轮询是另一个接口实现的，写出来是为了展示项目核心任务)
13. 前端定时轮询任务状态接口；
14. 后端优先查询 Redis；
15. 如果 Redis 查不到，说明任务未完成，返回 `pending`；
16. 如果 Redis 查到 `success`，返回 `success`；
17. 如果 Redis 查到 `failed`，返回 `failed`。

说明：MVP 中消费端不做完整实现，只需要知道它会消费 MQ、调用 AI 服务，并把结果写入 MySQL 和 Redis。

## 用户注册链路

流程如下：

1. 用户提交用户名和密码；
2. 后端校验参数；
3. 后端查询用户名是否已经存在；
4. 后端对密码进行哈希处理；
5. 后端写入 `users` 表；
6. 后端返回用户基本信息。

```text
用户提交注册信息
        |
参数校验
        |
查询用户名是否已存在
        |
密码哈希
        |
写入 users 表
        |
返回用户基本信息
```

## 用户登录链路

流程如下：

1. 用户提交用户名和密码；
2. 后端根据用户名查询用户；
3. 后端校验密码哈希；
4. 校验通过后生成 JWT；
5. 后端返回 JWT 和过期时间。

```text
用户提交登录信息
        |
根据用户名查询用户
        |
校验密码哈希
        |
生成 JWT
        |
返回 Token 和过期时间
```

## 提交任务流程

```text
用户提交 AI 任务
        |
校验 JWT
        |
IP 限流 + 用户限流
        |
创建 task_id
        |
任务写入 MySQL，状态 pending
        |
投递 MQ
        |
返回 task_id
```

## 后台消费任务流程(不做具体实现)

```text
消费端消费 MQ 队列任务
        |
调用 AI 服务
        |
成功？
  |-- 是：更新 MySQL 状态 success，写入 result
  |       |
  |       写入 Redis：task_id -> success
  |       |
  |       ACK MQ
  |
  |-- 否：重试
          |
          达到最大重试次数？
            |-- 否：继续重试
            |-- 是：更新 MySQL 状态 failed
                    |
                    写入 Redis：task_id -> failed
```

## 前端轮询流程

前端拿到 `task_id` 后，可以每隔 3 秒请求一次任务状态接口。

```text
前端每 3 秒请求任务状态接口
        |
后端校验 JWT
        |
校验任务归属
        |
查询 Redis
        |
Redis 是否存在状态？
  |-- 不存在：返回 pending
  |-- success：返回 success
  |-- failed：返回 failed
```
